# F1: Supabase Security Best Practices

Research date: 2026-07-13
Source: Official Supabase documentation

---

## Service Role Key & Secret Management

### [1] Service role key must never be exposed to the browser or clients
- quote: "Supabase provides special 'Service' keys, which can be used to bypass RLS. These should never be used in the browser or exposed to customers, but they are useful for administrative tasks."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [2] Service role key should only be used server-side in API routes
- quote: "service_role — For elevated access. This role is used by the API (PostgREST) to bypass Row Level Security."
- url: https://supabase.com/docs/guides/database/postgres/roles
- source_type: primary
- confidence: high

### [3] API keys should be stored in encrypted stores, never in code or client bundles
- quote: "You are also responsible for managing your database secrets and API keys, storing them safely in an encrypted store."
- url: https://supabase.com/docs/guides/platform/shared-responsibility-model
- source_type: primary
- confidence: high

### [4] Use publishable (anon) key for client-side, service_role for server-side only
- quote: "NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=supabase_publishable_key" — the `NEXT_PUBLIC_` prefix env var is for the anon key only, never the service_role key.
- url: https://supabase.com/docs/guides/auth/server-side/nextjs
- source_type: primary
- confidence: high

## RLS Policy Design Patterns

### [5] RLS must be enabled on ALL tables in exposed schemas — this is the #1 security rule
- quote: "RLS *must* always be enabled on any tables stored in an exposed schema. By default, this is the public schema."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [6] Tables without RLS allow any client to access and modify data freely
- quote: "Tables that do not have RLS enabled with reasonable policies allow any client to access and modify their data. This is usually not what you want."
- url: https://supabase.com/docs/guides/platform/going-into-prod
- source_type: primary
- confidence: high

### [7] Use `(select auth.uid())` instead of bare `auth.uid()` in policies — 95%+ performance improvement
- quote: "This method works well for JWT functions like auth.uid() and auth.jwt() as well as security definer Functions. Wrapping the function causes an initPlan to be run by the Postgres optimizer, which allows it to 'cache' the results per-statement."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [8] Always specify TO roles in policies to prevent unnecessary evaluation
- quote: "This prevents the policy ((select auth.uid()) = user_id) from running for any anon users, since the execution stops at the to authenticated step."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [9] For multi-tenant apps, store team membership in app_metadata (not user_metadata) for RLS use
- quote: "raw_user_meta_data — can be updated by the authenticated user using the supabase.auth.update() function. It is not a good place to store authorization data. raw_app_meta_data — cannot be updated by the user, so it's a good place to store authorization data."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [10] Use security definer functions for cross-table lookups in RLS to bypass join penalties
- quote: "A security definer function runs using the same role that created the function. This means that if you create a role with a superuser (like postgres), then that function will have bypassrls privileges."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [11] Use Custom Access Token Hooks for RBAC claims in JWT
- quote: "The Custom Access Token Auth Hook runs before a token is issued. You can use it to edit the JWT."
- url: https://supabase.com/docs/guides/api/custom-claims-and-role-based-access-control-rbac
- source_type: primary
- confidence: high

### [12] Explicitly check for auth.uid() IS NOT NULL in policies to handle unauthenticated users
- quote: "When a request is made without an authenticated user (e.g., no access token is provided or the session has expired), auth.uid() returns null. This means that a policy like USING (auth.uid() = user_id) will silently fail for unauthenticated users."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

## Supabase Auth Middleware Patterns (Next.js)

### [13] NEVER trust getSession() on server — always use getClaims() to verify JWT
- quote: "Always use supabase.auth.getClaims() to protect pages and user data. Never trust supabase.auth.getSession() inside server code such as Proxy. It isn't guaranteed to revalidate the Auth token."
- url: https://supabase.com/docs/guides/auth/server-side/nextjs
- source_type: primary
- confidence: high

### [14] Create new Supabase client per request on server (not module-level singleton)
- quote: "On the server, it basically configures a fetch call. You need to reconfigure the fetch call anew for every request to your server, because you need the cookies from the request."
- url: https://supabase.com/docs/guides/auth/server-side/nextjs
- source_type: primary
- confidence: high

### [15] Set Cache-Control: private, no-store on auth responses to prevent session leakage via CDN
- quote: "When @supabase/ssr refreshes a session token server-side, it writes the updated JWT to the HTTP response via a Set-Cookie header. If your CDN caches that response and serves it to a different user, that user's browser will store the cached token and be signed in as the wrong person."
- url: https://supabase.com/docs/guides/auth/server-side/advanced-guide
- source_type: primary
- confidence: high

### [16] Disable ISR on authenticated routes to prevent session token caching
- quote: "Do not enable ISR on any route where authentication is handled or where a session refresh can occur."
- url: https://supabase.com/docs/guides/auth/server-side/advanced-guide
- source_type: primary
- confidence: high

### [17] getClaims() validates JWT locally using WebCrypto + JWKS; getUser() makes network call
- quote: "getClaims reads the access token from storage and verifies it. Locally via the WebCrypto API and a cached JWKS endpoint when the project uses asymmetric signing keys."
- url: https://supabase.com/docs/guides/auth/server-side/nextjs
- source_type: primary
- confidence: high

### [18] Vercel Fluid Compute can leak sessions if client is initialized at module scope
- quote: "Vercel's Fluid compute model can keep server instances warm and reuse them across requests. A Supabase client initialized in module scope — or stored in a shared variable — may be reused across requests from different users, causing one user's session to leak."
- url: https://supabase.com/docs/guides/auth/server-side/advanced-guide
- source_type: primary
- confidence: high

## Common Supabase Security Pitfalls

### [19] Forgetting to enable RLS on tables created via SQL (Dashboard auto-enables, SQL does not)
- quote: "RLS is enabled by default on tables created with the Table Editor in the dashboard. If you create one in raw SQL or with the SQL editor, remember to enable RLS yourself."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [20] Using user_metadata in RLS policies (users can modify it themselves)
- quote: "Not all information present in the JWT should be used in RLS policies. For instance, creating an RLS policy that relies on the user_metadata claim can create security issues in your application as this information can be modified by authenticated end users."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [21] JWT is not always fresh — changes to app_metadata aren't reflected until token refresh
- quote: "A JWT is not always 'fresh'. Even if you remove a user from a team and update the app_metadata field, that will not be reflected using auth.jwt() until the user's JWT is refreshed."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [22] Views bypass RLS by default (security_definer) — use security_invoker in Postgres 15+
- quote: "Views bypass RLS by default because they are usually created with the postgres user. This is a feature of Postgres, which automatically creates views with security definer. In Postgres 15 and above, you can make a view obey the RLS policies of the underlying tables by setting security_invoker = true."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [23] Security-definer functions must NOT be created in exposed schemas
- quote: "Security-definer functions should never be created in a schema in the 'Exposed schemas' inside your API settings."
- url: https://supabase.com/docs/guides/auth/row-level-security
- source_type: primary
- confidence: high

### [24] Email link tracking can break Supabase auth links — disable link tracking in SMTP
- quote: "When using a custom SMTP service, some services might have link tracking enabled which may overwrite or deform the email confirmation links sent by Supabase Auth. To prevent this from happening, we recommend that you disable link tracking when using a custom SMTP service."
- url: https://supabase.com/docs/guides/platform/going-into-prod
- source_type: primary
- confidence: high

### [25] Database is shared responsibility — Supabase won't enforce RLS for you
- quote: "Running databases is a shared responsibility between you and Supabase. We give you full access to the database. You are responsible for ensuring that tables with sensitive data have the right level of access."
- url: https://supabase.com/docs/guides/platform/shared-responsibility-model
- source_type: primary
- confidence: high

## Summary of Key Recommendations for CrewWork

1. **Service role key**: Only use in Next.js API routes / server actions, never in `NEXT_PUBLIC_` env vars or client components
2. **RLS**: Enable on ALL tables. Use `(select auth.uid())` wrapped in SELECT for performance. Always specify `TO authenticated` or `TO anon` roles
3. **Auth middleware**: Use `getClaims()` (not `getSession()`) in Next.js middleware/proxy. Create fresh client per request
4. **Multi-tenant**: Use `app_metadata` for team/role claims (not `user_metadata`). Consider Custom Access Token Hook for RBAC
5. **CDN/ISR**: Set `Cache-Control: private, no-store` on auth responses. Disable ISR on authenticated routes
6. **Anti-patterns to avoid**: Don't use user_metadata in RLS, don't create security-definer functions in exposed schemas, don't trust getSession() server-side
