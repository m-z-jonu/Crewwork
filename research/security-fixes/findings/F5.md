# F5: Cryptographic Key Management — Key Rotation, Prekey Pools, and Secure Storage for Messaging Apps

**Research Date**: 2026-07-13
**Sources**: Signal Protocol specs (X3DH, PQXDH, Double Ratchet), Matrix Client-Server API, MDN Web Crypto API

---

## 1. Identity Key Rotation Strategies

### [1] Identity keys are long-term and rarely rotated — rotation triggers a new trust establishment
- quote: "Each party has a long-term identity public key (*IKA* for Alice, *IKB* for Bob). Bob only needs to upload his identity key to the server once."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [2] Identity key compromise allows permanent impersonation — this is why prekeys exist as a buffer layer
- quote: "Compromise of a party's identity private key allows impersonation of that party to others. Compromise of a party's prekey private keys may affect the security of older or newer *SK* values, depending on many considerations."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [3] Identity key rotation should be a last resort — prekeys and ratcheting provide forward secrecy without rotating identity
- quote: "Frequent replacement of signed prekeys mitigates this, as does using a post-X3DH ratcheting protocol which rapidly replaces *SK* with new keys to provide fresh forward secrecy."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [4] In practice, rotate identity keys only when compromise is suspected or during major account migration
- quote: "Bob only needs to upload his identity key to the server once."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [5] Matrix uses device-level identity keys with cross-signing for identity verification
- quote: Matrix spec defines device keys, cross-signing keys (master, self-signing, user-signing), and key verification framework for establishing trust in identity.
- url: https://matrix.org/v1.19/client-server-api/#end-to-end-encryption
- source_type: primary
- confidence: high

## 2. Prekey Pool Replenishment

### [6] Signed prekeys should be rotated on a regular schedule (weekly to monthly)
- quote: "Bob will also upload a new signed prekey and prekey signature at some interval (e.g. once a week, or once a month). The new signed prekey and prekey signature will replace the previous values."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [7] One-time prekeys are consumed on use — server should notify client when pool is running low
- quote: "Bob may upload new one-time prekeys at other times (e.g. when the server informs Bob that the server's store of one-time prekeys is getting low)."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [8] After consuming a one-time prekey, the private key MUST be deleted for forward secrecy
- quote: "Bob deletes any one-time prekey private key that was used, for forward secrecy."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [9] Old signed prekey private keys should be retained briefly for delayed messages, then deleted
- quote: "After uploading a new signed prekey, Bob may keep the private key corresponding to the previous signed prekey around for some period of time, to handle messages using it that have been delayed in transit. Eventually, Bob should delete this private key for forward secrecy."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [10] Server must rate-limit prekey bundle fetching to prevent one-time prekey draining attacks
- quote: "One party maliciously drains another party's one-time prekeys, so the server should attempt to prevent this, e.g. with rate limits on fetching prekey bundles."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [11] PQXDH adds a "last-resort" post-quantum prekey as fallback when one-time PQ prekeys are exhausted
- quote: "Bob has a signed last-resort *pqkem* prekey *PQSPKB*, which he changes periodically and signs each time with *IKB*, and a set of signed one-time prekeys... The name 'last-resort' refers to the fact that the last-resort prekey is only used when one-time *pqkem* prekeys are not available."
- url: https://signal.org/docs/specifications/pqxdh/
- source_type: primary
- confidence: high

### [12] Matrix uses the same pattern: one-time keys + fallback keys for async key agreement
- quote: Matrix spec defines "One-time and fallback keys" as the mechanism for Olm key agreement — clients upload one-time keys and the server hands them out one at a time, similar to Signal's prekey pool.
- url: https://matrix.org/v1.19/client-server-api/#one-time-and-fallback-keys
- source_type: primary
- confidence: high

### [13] Recommended prekey pool size: 20-100 one-time prekeys, with server-side threshold trigger for refill at ~10 remaining
- quote: From Signal protocol analysis — Bob publishes "a set of Bob's one-time prekeys (*OPKB1*, *OPKB2*, *OPKB3*, ...)" — practical implementations typically maintain 20-100 prekeys with threshold-based replenishment.
- url: https://signal.org/docs/specifications/x3dh/
- source_type: inferred
- confidence: medium

## 3. Key Compromise Recovery

### [14] Identity key compromise = full impersonation — recovery requires generating new identity key and re-establishing trust
- quote: "Compromise of a party's identity private key allows impersonation of that party to others."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [15] Signed prekey compromise can be recovered by rotating to a new signed prekey on the server
- quote: "Compromise of prekey private keys may enable attacks that extend into the future, such as passive calculation of *SK* values, and impersonation of arbitrary other parties... These attacks are possible until the compromised party replaces his compromised prekeys on the server (in the case of passive attack); or deletes his compromised signed prekey's private key (in the case of key-compromise impersonation)."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [16] One-time prekey compromise is limited in scope — each key is used once then deleted
- quote: "If one-time prekeys are used for a protocol run then a compromise of Bob's identity key and prekey private keys at some future time will not compromise the older *SK*, assuming the private key for *OPKB* was deleted."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

### [17] Double Ratchet provides break-in recovery — future messages become secure after one DH ratchet step
- quote: "Break-in recovery: Future output keys appear random to an adversary who learns the KDF key at some point in time, provided that future inputs have added sufficient entropy."
- url: https://signal.org/docs/specifications/doubleratchet/
- source_type: primary
- confidence: high

### [18] Recovery from compromise requires both parties to exchange a new DH ratchet step
- quote: "An eavesdropper who briefly compromises one of the parties might learn the value of a current ratchet private key, but that private key will eventually be replaced with an uncompromised one. At that point, the Diffie-Hellman calculation between ratchet key pairs will define a DH output unknown to the attacker."
- url: https://signal.org/docs/specifications/doubleratchet/
- source_type: primary
- confidence: high

### [19] Matrix cross-signing allows recovery — re-verify devices after rotating identity keys
- quote: Matrix spec defines cross-signing keys (master, self-signing, user-signing) that enable establishing trust after key rotation via SAS verification, QR codes, and key verification framework.
- url: https://matrix.org/v1.19/client-server-api/#cross-signing
- source_type: primary
- confidence: high

## 4. Secure Key Storage in Browsers (IndexedDB, WebCrypto)

### [20] CryptoKey objects are serializable and should be stored in IndexedDB — never expose raw key material to JS
- quote: "CryptoKey is a serializable object, which allows keys to be stored and retrieved using standard web storage APIs. The specification expects that most developers will use the IndexedDB API, storing CryptoKey objects against some key string identifier... This allows the storage and retrieval of the CryptoKey without having to expose its underlying key material to the application or the JavaScript environment."
- url: https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto
- source_type: primary
- confidence: high

### [21] Use wrapKey/unwrapKey to encrypt keys at rest in IndexedDB with a wrapping key
- quote: "If the key is sensitive you should use wrapKey(), which exports the key and then encrypts it using another key... The inverse of wrapKey() is unwrapKey(), which decrypts then imports the key."
- url: https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto
- source_type: primary
- confidence: high

### [22] WebCrypto supports X25519, ECDH, HKDF — all primitives needed for Signal Protocol
- quote: SubtleCrypto supports: ECDH (deriveKey), X25519 (deriveKey), HKDF (deriveKey, deriveBits), ECDSA, Ed25519, AES-GCM, SHA-256/384/512.
- url: https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto
- source_type: primary
- confidence: high

### [23] SubtleCrypto is only available in secure contexts (HTTPS) and Web Workers
- quote: "Secure context: This feature is available only in secure contexts (HTTPS). This feature is available in Web Workers."
- url: https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto
- source_type: primary
- confidence: high

### [24] Generate keys with extractable=false to prevent JavaScript from reading raw key bytes
- quote: generateKey() accepts an extractability parameter — when false, the CryptoKey cannot be exported or wrapped, keeping raw material inaccessible to JS.
- url: https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto/generateKey
- source_type: primary
- confidence: high

### [25] Best practice: generate identity key with extractable=false, store CryptoKey in IndexedDB, wrap prekeys with AES-GCM wrapping key derived from user passphrase
- quote: From WebCrypto best practices: generate non-extractable CryptoKey for long-term identity, use deriveKey with PBKDF2 or HKDF to derive wrapping keys from passphrases, wrap/unwrap prekeys for at-rest protection in IndexedDB.
- url: https://developer.mozilla.org/en-US/docs/Web/API/SubtleCrypto
- source_type: inferred
- confidence: medium

## 5. Key Management in Signal, WhatsApp, Matrix

### Signal Protocol Key Hierarchy

#### [26] Signal key hierarchy: Identity Key → Signed Prekey → One-Time Prekeys → Ephemeral Key (per session)
- quote: "X3DH uses the following elliptic curve public keys: *IKA* (Alice's identity key), *EKA* (Alice's ephemeral key), *IKB* (Bob's identity key), *SPKB* (Bob's signed prekey), *OPKB* (Bob's one-time prekey)."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- confidence: high

#### [27] Signal's Double Ratchet continuously derives new encryption keys — compromise of one key doesn't compromise others
- quote: "The parties derive new keys for every Double Ratchet message so that earlier keys cannot be calculated from later ones. The parties also send Diffie-Hellman public values attached to their messages."
- url: https://signal.org/docs/specifications/doubleratchet/
- source_type: primary
- confidence: high

#### [28] Signal Sesame manages sessions across multiple devices per account
- quote: "Sesame describes the Sesame algorithm for managing message encryption sessions in an asynchronous and multi-device setting."
- url: https://signal.org/docs/
- source_type: primary
- confidence: high

### WhatsApp Key Management (Based on Signal Protocol)

#### [29] WhatsApp uses Signal Protocol with server-side key storage for multi-device support
- quote: WhatsApp's multi-device architecture stores encrypted identity keys and session data on their servers, allowing up to 4 linked devices per account. Each device runs its own Signal Protocol session.
- url: https://faq.whatsapp.com/531873499549942
- source_type: secondary
- confidence: medium

#### [30] WhatsApp implements "key transparent" protocol for public key directory verification
- quote: WhatsApp uses a Key Transparency protocol (based on Apple's PERC and Google's VRF) to provide verifiable public key directories, preventing server-side identity key substitution attacks.
- url: https://blog.whatsapp.com/end-to-end-backups-and-the-future-of-whatsapp-security
- source_type: secondary
- confidence: medium

### Matrix Key Management

#### [31] Matrix uses Olm (1:1) and Megolm (group) ratcheting protocols with device-level identity keys
- quote: Matrix defines Olm for 1:1 encrypted sessions and Megolm for group encryption. Each device has its own Ed25519 signing key and Curve25519 identity key.
- url: https://matrix.org/v1.19/olm-megolm/
- source_type: primary
- confidence: high

#### [32] Matrix clients upload one-time keys and fallback keys to the server for async Olm session establishment
- quote: Matrix spec: "One-time and fallback keys" — clients upload ed25519 signing keys and curve25519 one-time keys; the server allocates them to establishing sessions.
- url: https://matrix.org/v1.19/client-server-api/#one-time-and-fallback-keys
- source_type: primary
- confidence: high

#### [33] Matrix cross-signing allows users to verify their own devices and other users' devices
- quote: Matrix defines master key, self-signing key, and user-signing key for cross-signing. Users can verify devices via SAS, QR codes, or emoji verification.
- url: https://matrix.org/v1.19/client-server-api/#cross-signing
- source_type: primary
- confidence: high

#### [34] Matrix server-side key backup encrypts Megolm session keys with a key derived from a recovery key
- quote: Matrix spec defines "Server-side key backups" using `m.megolm_backup.v1.curve25519-aes-sha2` — backup keys are encrypted with a key derived from a user-provided recovery key (passphrase or 48-character code).
- url: https://matrix.org/v1.19/client-server-api/#server-side-key-backups
- source_type: primary
- confidence: high

## 6. Recommended Key Rotation Strategy for CrewWork

### [35] Identity key: generate once, store non-extractable in IndexedDB, rotate only on explicit user action or compromise
- Based on Signal's design: identity keys are long-term; rotation triggers re-establishment of all trust relationships.

### [36] Signed prekeys: rotate weekly/monthly, keep old private key for 1 week then delete
- Based on Signal X3DH: "once a week, or once a month" replacement with grace period for delayed messages.

### [37] One-time prekeys: maintain pool of 50-100, server triggers refill when <10 remaining, delete private keys immediately after use
- Based on Signal protocol pattern and best practices.

### [38] Prekey pool refill: client checks on startup + periodically, uploads batch of 50 new prekeys when threshold hit
- Derived from Signal's pattern of server-notifying client when pool is low.

### [39] Use HKDF (via WebCrypto) for all key derivation — separate keys for signing, encryption, and session establishment
- Based on Signal's use of HKDF in KDF_RK and KDF_CK functions.

### [40] For post-quantum readiness: add ML-KEM/Kyber last-resort prekey that is rotated on same schedule as signed prekey
- Based on Signal's PQXDH specification.
