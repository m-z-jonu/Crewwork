**Research date:** 2026-07-13
**Angle:** How to properly verify signed prekeys in X3DH, implement key rotation, and handle key compromise

## 1. Signed Prekey Verification in X3DH

### The Core Mechanism

In X3DH, Bob publishes a **prekey bundle** containing:
- Identity key (`IKB`)
- Signed prekey (`SPKB`)
- Prekey signature: `Sig(IKB, Encode(SPKB))`
- One-time prekeys (`OPKB1`, `OPKB2`, ...)

**Alice's verification obligation** (X3DH §3.3): Alice MUST verify the prekey signature before proceeding. If verification fails, she aborts the protocol entirely.

```
Alice verifies: Verify(IKB, Encode(SPKB), Sig(IKB, Encode(SPKB))) == true
```

### Why Verification is Critical — The "Weak Forward Secrecy" Attack

From X3DH §4.5 (Signatures):

> "It might be tempting to omit the prekey signature after observing that mutual authentication and forward secrecy are achieved by the DH calculations. However, this would allow a **'weak forward secrecy' attack**: A malicious server could provide Alice a prekey bundle with **forged prekeys**, and later compromise Bob's `IKB` to calculate `SK`."

Without signature verification:
1. Server intercepts Alice's prekey bundle request
2. Server generates its own forged `SPKB'` and `OPKB'`
3. Alice computes SK using the forged keys
4. Server later compromises Bob's identity key `IKB`
5. Server can now compute the same SK and decrypt all messages

**Attack scenario:** Server replaces Bob's signed prekey with an attacker-controlled key, then later compromises Bob's identity key to decrypt the session.

### libsignal Implementation Reference

From `libsignal/rust/protocol/src/session.rs`, the `process_prekey_bundle` function:

```rust
// Verify signed prekey signature
if !their_identity_key.public_key().verify_signature(
    &bundle.signed_pre_key_public()?.serialize(),
    bundle.signed_pre_key_signature()?,
) {
    return Err(SignalProtocolError::SignatureValidationFailed);
}

// Verify kyber prekey signature (PQXDH)
if !their_identity_key.public_key().verify_signature(
    &bundle.kyber_pre_key_public()?.serialize(),
    bundle.kyber_pre_key_signature()?,
) {
    return Err(SignalProtocolError::SignatureValidationFailed);
}
```

Key points from libsignal:
- Identity trust is checked first via `is_trusted_identity()`
- Both signed EC prekey AND kyber prekey signatures must verify
- Failure returns `SignatureValidationFailed` — abort the protocol
- After successful verification, identity key is saved to the store

### PQXDH Extension

PQXDH (§3.2) requires signatures on ALL prekeys:
- Signed EC prekey: `Sig(IKB, EncodeEC(SPKB), ZSPK)`
- Signed last-resort PQ prekey: `Sig(IKB, EncodeKEM(PQSPKB), ZPQSPK)`
- Signed one-time PQ prekeys: `Sig(IKB, EncodeKEM(PQOPKBn), Zn)`

Alice verifies ALL signatures before proceeding (PQXDH §3.3):

> "Alice verifies the signatures on the prekeys. If any signature check fails, Alice aborts the protocol."

### Verification Implementation Checklist

For CrewWork to properly verify signed prekeys:

1. **On receiving a prekey bundle from server:**
   - Extract Bob's identity key `IKB`
   - Extract signed prekey `SPKB`
   - Extract signature `Sig(IKB, Encode(SPKB))`
   - Compute: `Verify(IKB, SPKB_encoded, signature)`
   - If false → ABORT, do not proceed with key agreement

2. **Signature scheme:** XEdDSA (for Curve25519/Curve448 keys)
   - Sign with Ed25519 key derived from X25519 key
   - Domain separation via `0xFF` bytes in HKDF

3. **Key encoding:** Single-byte curve identifier + little-endian u-coordinate

4. **Store trust verification:**
   - Check if identity key is already trusted
   - If new/changed identity key → prompt user to verify (QR code, fingerprint)
   - If identity key changed unexpectedly → security warning

## 2. Key Rotation Best Practices

### Signed Prekey Rotation

From X3DH §3.2 and PQXDH §3.2:

> "Bob will also upload a new signed prekey and prekey signature at some interval (e.g. **once a week**, or **once a month**). The new signed prekey and prekey signature will replace the previous values."

**Rotation strategy:**
- **Frequency:** Weekly to monthly (Signal uses weekly)
- **Overlap period:** Keep old signed prekey private key for `MAXLATENCY` period after rotation to handle delayed messages
- **Deletion:** Delete old signed prekey private key after overlap period for forward secrecy

### One-Time Prekey Management

From X3DH §3.2:

> "Bob may upload new one-time prekeys at other times (e.g. when the server informs Bob that the server's store of one-time prekeys is getting low)."

**Best practices:**
- Maintain a pool of 100+ one-time prekeys
- Monitor server's prekey count
- Upload new batch when count drops below threshold (e.g., 10)
- One-time prekeys are deleted from server after use (single-use)

### Session Expiration (Sesame)

From Sesame §4.2:

> "It may be desirable for devices to periodically replace old sessions with new sessions, for security purposes."

**Time constants:**
- `MAXSEND`: Time after which a session must no longer be used for encryption
- `MAXRECV`: Time after which a session may be deleted (must be > `MAXSEND + 2*MAXLATENCY`)
- Sessions are moved to inactive list after `MAXSEND`
- Sessions are deleted after `MAXRECV` (after fetching all mailbox messages)

### Post-Quantum Key Rotation (PQXDH)

PQXDH adds rotation requirements for PQ KEM prekeys:
- Rotate last-resort PQ prekey periodically (like signed prekeys)
- Rotate signed one-time PQ prekeys
- Delete used one-time PQ prekeys from server

### Implementation Recommendations for CrewWork

1. **Signed prekey rotation schedule:**
   ```python
   # Rotate signed prekeys every 7 days
   SIGNED_PREKEY_ROTATION_INTERVAL = 7 * 24 * 60 * 60  # seconds
   
   # Keep old prekey for delayed messages
   OLD_PREKEY_RETENTION = 2 * MAX_LATENCY  # e.g., 2 * 7 days
   ```

2. **One-time prekey pool management:**
   ```python
   MIN_PREKEY_POOL_SIZE = 10
   MAX_PREKEY_POOL_SIZE = 100
   PREKEY_UPLOAD_BATCH_SIZE = 50
   
   # Server notifies when pool is low
   if prekey_count < MIN_PREKEY_POOL_SIZE:
       upload_new_prekeys(PREKEY_UPLOAD_BATCH_SIZE)
   ```

3. **Session expiration policy:**
   ```python
   MAX_SEND_AGE = 30 * 24 * 60 * 60  # 30 days
   MAX_RECV_AGE = MAX_SEND_AGE + 2 * MAX_LATENCY
   ```

4. **Server-side prekey management:**
   - Track prekey usage per device
   - Notify devices when prekey pool is low
   - Rate limit prekey bundle requests (prevent prekey draining attacks)
   - Delete one-time prekeys immediately after use

## 3. Key Compromise Recovery Mechanisms

### Compromise Scenarios and Impact

From X3DH §4.6 and PQXDH §4.6:

**Identity Key (`IKB`) Compromise:**
- Allows **impersonation** of Bob to others
- Permanent damage until identity key is replaced
- All future sessions with Bob are compromised
- **Recovery:** Generate new identity key pair, notify all contacts

**Signed Prekey (`SPKB`) Compromise:**
- If one-time prekey was used: older `SK` is safe (forward secrecy maintained)
- If one-time prekey was NOT used: `SK` from that session is compromised
- **Recovery:** Delete compromised signed prekey, upload new one
- **Mitigation:** Frequent rotation limits exposure window

**One-Time Prekey (`OPKB`) Compromise:**
- If deleted from server after use: older `SK` is safe
- If still on server: server could serve it to attacker
- **Recovery:** Server deletes prekey after use; if compromised, upload replacement

### Forward Secrecy and Break-in Recovery

From Double Ratchet §2.1:

> "KDF chain properties:
> - **Forward security:** Output keys from the past appear random to an adversary who learns the KDF key at some point in time.
> - **Break-in recovery:** Future output keys appear random to an adversary who learns the KDF key at some point in time, provided that future inputs have added sufficient entropy."

The Double Ratchet provides **break-in recovery** through DH ratchet steps:
- Each new ratchet key pair introduces fresh entropy
- After compromise, the attacker is locked out once both parties perform a DH ratchet step
- Compromise of sending/receiving chain keys only affects current and future messages until next DH ratchet

### Recovery Process

From Sesame §6.2:

> "Security is catastrophically compromised if an attacker learns a device's secret values, such as the identity private key and session state. Recovery from a device compromise requires the user to **replace the compromised device and compromised identity key pair and notify all correspondents of the new public key**."

**Full recovery procedure:**

1. **Detect compromise** (or assume compromise after suspicious activity)
2. **Generate new identity key pair:**
   ```
   new_IK = GenerateKeyPair()
   ```
3. **Upload new identity key to server**
4. **Notify all contacts** of new identity key
5. **Contacts must verify** new identity key through authenticated channel
6. **Delete all old sessions** with compromised device
7. **Start fresh sessions** with new identity key

### Mitigation Strategies

From Sesame §6.2:

> "To mitigate [key-compromise impersonation], Sesame devices using X3DH and the Double Ratchet should **delete their signed prekeys on a regular basis**, without allowing a malicious server to inhibit deletion."

**Pre-compromise mitigations:**
1. Frequent signed prekey rotation (weekly)
2. Use one-time prekeys for all new sessions
3. Enable session expiration
4. Server-side rate limiting on prekey bundle requests

**Post-compromise mitigations:**
1. Immediate identity key replacement
2. Contact notification ("safety number changed" warnings)
3. Session expiration clears lingering compromised sessions
4. Double Ratchet break-in recovery limits ongoing damage

### Key Compromise Impersonation (KCI)

From Sesame §6.2:

> "An attacker might try to perform **key-compromise impersonation**, where the attacker impersonates other parties to the compromised party... This attacker capability would continue until the target deletes their compromised signed prekey private key(s)."

**KCI scenario:**
1. Attacker compromises Bob's signed prekey
2. Server uses compromised prekey to create fake sessions with Bob
3. Bob receives messages that appear to come from Alice but actually come from attacker
4. This continues until Bob deletes the compromised signed prekey

**Defense:**
- Delete signed prekeys on schedule (not server-controlled)
- Use session expiration to clear attacker-created sessions
- Verify safety numbers regularly

## 4. Signal's Approach to Key Management

### Key Hierarchy

```
Identity Key (IK)
  └── Signs: Signed Prekey, One-Time Prekeys, PQ Prekeys
  └── Long-lived, rarely rotated
  
Signed Prekey (SPK)
  └── Used in X3DH/PQXDH for session initialization
  └── Rotated weekly/monthly
  
One-Time Prekey (OPK)
  └── Single-use, deleted after use
  └── Pool of 100+ keys
  
Ephemeral Key (EK)
  └── Generated fresh per X3DH/PQXDH session
  └── Deleted immediately after use
```

### Session Lifecycle

1. **Prekey publication:** Bob publishes IK, SPK (signed), OPKs to server
2. **Session initiation:** Alice fetches prekey bundle, verifies signatures, computes SK
3. **Initial message:** Alice sends X3DH/PQXDH initial message with identity key
4. **Double Ratchet:** Both parties ratchet keys for ongoing communication
5. **Session expiration:** Sessions are expired and replaced periodically

### Authentication Methods

From X3DH §4.1:

> "Before or after an X3DH key agreement, the parties may compare their identity public keys `IKA` and `IKB` through some authenticated channel. For example, they may compare public key fingerprints manually, or by scanning a QR code."

**Signal's methods:**
- QR code scanning (in-person verification)
- Safety number comparison (numeric fingerprint)
- Key transparency (future: public key directory with transparency logs)

### Post-Quantum Migration

Signal has migrated from X3DH to PQXDH:
- Added ML-KEM (Kyber) post-quantum KEM prekeys
- Signed PQ prekeys with identity key
- Hybrid approach: classical + post-quantum
- Double Ratchet with PQ ratchet (SPQR) in development

1. **Verify signed prekey signature:**

   // Before computing X3DH shared secret
   if !their_identity_key.verify_signature(signed_prekey, signature) {

## Summary of Findings

### Critical Finding: Signed Prekey Verification is MANDATORY

The X3DH specification explicitly states that Alice MUST verify the signed prekey signature. Without this verification:
- A malicious server can perform a "weak forward secrecy" attack
- Server can forge prekeys and later compromise the session
- This completely undermines the security model of X3DH

### Key Rotation Best Practices

- **Signed prekeys:** Rotate weekly to monthly
- **One-time prekeys:** Maintain pool of 100+, replenish when low
- **Session expiration:** Implement MAX_SEND_AGE (30 days) and MAX_RECV_AGE
- **Overlap period:** Keep old signed prekey for delayed messages, then delete

### Compromise Recovery

- **Identity key compromise:** Replace key, notify all contacts, start fresh
- **Signed prekey compromise:** Delete compromised key, upload new one
- **Break-in recovery:** Double Ratchet provides this through DH ratchet steps
- **Prevention:** Frequent rotation, one-time prekeys, session expiration

### Implementation Priority

1. **Immediate:** Add signed prekey signature verification
2. **Short-term:** Implement key rotation for signed prekeys
3. **Medium-term:** Add one-time prekey pool management
4. **Long-term:** Implement session expiration and compromise recovery
