# F4: React Security Patterns — Safe Alternatives to dangerouslySetInnerHTML

**Research Date:** 2026-07-12
**Focus:** Safe SVG rendering in React, DOMPurify usage, React security anti-patterns, CSP for React apps

---

## 1. The Risk: dangerouslySetInnerHTML in React

`dangerouslySetInnerHTML` sets the `innerHTML` property of a DOM node, bypassing React's virtual DOM entirely. React's own documentation explicitly warns:

> "If the HTML inside isn't trusted (for example, if it's based on user data), you risk introducing an XSS vulnerability."

**Source:** https://react.dev/reference/react-dom/components/common#dangerouslysettinginnerhtml (React docs, current)

When SVG is rendered via `dangerouslySetInnerHTML`, the SVG can contain:
- `<script>` tags (though browsers generally don't execute them inside SVG `<foreignObject>`)
- `<a xlink:href="javascript:...">` — JavaScript URL schemes in SVG links
- Event handlers on SVG elements (`onload`, `onclick`, etc.)
- `<use>` + `<animate>` combinations for mutation XSS
- `<foreignObject>` containing arbitrary HTML

## 2. Safe Alternatives to dangerouslySetInnerHTML

### 2a. React JSX (Preferred)

React's JSX auto-escapes all content. SVG rendered as JSX is inherently safe from XSS:

```jsx
// SAFE - React escapes all string values
<svg viewBox="0 0 24 24">
  <path d={svgPathData} fill={color} />
</svg>
```

**When SVG data comes as a string, parse it into JSX components** using a library like `react-svg` or `@svgr/react`:

```jsx
import { ReactSVG } from 'react-svg';
<ReactSVG src="/icon.svg" />

// OR with SVGR (converts SVG files to React components at build time)
import { ReactComponent as MyIcon } from './icon.svg';
<MyIcon />
```

### 2b. DOMPurify Sanitization (When HTML strings are unavoidable)

DOMPurify (v3.4.12, by Cure53) is the OWASP-recommended HTML sanitizer. It supports HTML, SVG, and MathML.

**Basic React pattern:**

```jsx
import DOMPurify from 'dompurify';

function SafeSvgComponent({ svgString }) {
  const clean = DOMPurify.sanitize(svgString, {
    USE_PROFILES: { svg: true },
    ADD_TAGS: ['use'],           // if SVG references external defs
    ADD_ATTR: ['xlink:href'],    // if SVG uses xlink
  });

  return (
    <div dangerouslySetInnerHTML={{ __html: clean }} />
  );
}
```

**Key DOMPurify config for SVG:**

| Config | Value | Purpose |
|--------|-------|---------|
| `USE_PROFILES` | `{ svg: true }` | Allow only safe SVG elements |
| `USE_PROFILES` | `{ html: true }` | HTML-only (no SVG, no MathML) |
| `ADD_TAGS` | `['use', 'foreignObject']` | Extend tag allowlist as needed |
| `ADD_ATTR` | `['xlink:href', 'clip-path']` | Extend attribute allowlist |
| `FORBID_TAGS` | `['style', 'script']` | Block specific dangerous tags |
| `RETURN_TRUSTED_TYPE` | `true` | Returns TrustedHTML (for Trusted Types) |

**Server-side (Node.js) pattern:**

```js
import { JSDOM } from 'jsdom';
import DOMPurify from 'dompurify';

const window = new JSDOM('').window;
const purify = DOMPurify(window);
const clean = purify.sanitize(svgString, { USE_PROFILES: { svg: true } });
```

**Critical caveat from DOMPurify docs:** "If you first sanitize HTML and then modify it afterwards, you might easily void the effects of sanitization." Do not post-process sanitized output.

**Source:** https://github.com/cure53/DOMPurify (DOMPurify README, v3.4.12)

### 2c. Browser HTML Sanitizer API (Emerging)

The HTML Sanitizer API (inspired by DOMPurify) is shipping in Chrome/Edge and being standardized. DOMPurify is the reference implementation.

```js
const sanitizer = new Sanitizer();
el.setHTML(svgString, { sanitizer });
```

**Browser support:** Chrome 83+, Edge 83+, Firefox behind flag. Not yet widely available.

## 3. React Security Anti-patterns to Avoid

From OWASP XSS Prevention Cheat Sheet (current):

| Anti-pattern | Why it's dangerous |
|---|---|
| `dangerouslySetInnerHTML` without sanitization | Direct XSS vector — raw HTML/JS execution |
| `javascript:` or `data:` URLs in `href`/`src` | React does NOT auto-block these — requires manual validation |
| Using `eval()`, `setTimeout(string)`, `new Function()` | Code injection sinks |
| Rendering user-supplied SVG without DOMPurify | SVG can contain `<script>`, event handlers, `xlink:href` JS |
| Sanitizing then modifying the output | Post-sanitization modification voids protection |
| Using `innerHTML` directly (bypassing React) | Skips React's entire security model |

**Source:** https://cheatsheetseries.owasp.org/cheatsheets/Cross_Site_Scripting_Prevention_Cheat_Sheet.html (OWASP, current)

## 4. Content Security Policy (CSP) for React Apps

CSP is a defense-in-depth layer, NOT a primary XSS prevention mechanism.

### Recommended CSP headers for a React app:

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-{random}';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  svg-src 'self';
  object-src 'none';
  base-uri 'self';
  form-action 'self';
  frame-ancestors 'none';
  require-trusted-types-for 'script';
  report-uri /csp-report-endpoint
```

### Key CSP directives relevant to SVG/XSS:

| Directive | Purpose |
|---|---|
| `object-src 'none'` | Blocks `<embed>`, `<object>`, `<applet>` |
| `script-src 'self' 'nonce-...'` | Blocks inline scripts (including in SVG) |
| `require-trusted-types-for 'script'` | Forces Trusted Types for all DOM XSS sinks |
| `svg-src 'self'` | Restricts SVG loading origins |
| `base-uri 'self'` | Prevents `<base>` tag injection |

**Source:** https://cheatsheetseries.owasp.org/cheatsheets/Content_Security_Policy_Cheat_Sheet.html (OWASP CSP Cheat Sheet)

### Trusted Types (Chromium)

Trusted Types are one of the few controls that **eliminate** entire classes of DOM XSS rather than just mitigating them.

```html
Content-Security-Policy: require-trusted-types-for 'script'
```

This causes all DOM XSS sinks (`innerHTML`, `outerHTML`, `document.write`, `script.src`) to **reject plain strings**. Every assignment must go through a vetted policy. DOMPurify supports Trusted Types natively:

```js
el.innerHTML = DOMPurify.sanitize(html, { RETURN_TRUSTED_TYPE: true });
```

**Source:** https://web.dev/articles/trusted-types (Google web.dev, Krzysztof Kotowicz)

## 5. Recommended Fix Strategy for CrewWork

For the `dangerouslySetInnerHTML` SVG rendering in CrewWork:

1. **Immediate fix:** Wrap all SVG strings through DOMPurify before rendering:
   ```jsx
   const cleanSvg = DOMPurify.sanitize(rawSvg, {
     USE_PROFILES: { svg: true },
     RETURN_TRUSTED_TYPE: true
   });
   return <div dangerouslySetInnerHTML={{ __html: cleanSvg }} />;
   ```

2. **Better long-term:** Replace `dangerouslySetInnerHTML` with a dedicated SVG component that parses SVG into JSX using SVGR or a DOMParser-based approach.

3. **Defense-in-depth:** Deploy a CSP header with `require-trusted-types-for 'script'` and `object-src 'none'`.

4. **Never sanitize then modify** — sanitize at the boundary, render immediately.
