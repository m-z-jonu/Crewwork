**Research Date**: 2026-07-13
**Sources**: Official Next.js v16.2.10 documentation, Next.js security blog

## 1. Authentication Middleware Patterns (Proxy in v16)

> In Next.js 16, `middleware.ts` has been **deprecated and renamed to `proxy.ts`**. The function must be named `proxy` (not `middleware`).

### Pattern A: Proxy-level authentication (optimistic checks)

Use Proxy for quick redirects based on session cookies. This runs on every route before rendering.

```ts
// proxy.ts
import { NextRequest, NextResponse } from 'next/server'
import { decrypt } from '@/app/lib/session'

const protectedRoutes = ['/dashboard', '/admin']
const publicRoutes = ['/login', '/signup', '/']

export default async function proxy(req: NextRequest) {
  const path = req.nextUrl.pathname
  const isProtectedRoute = protectedRoutes.includes(path)

  const session = await decrypt(req.cookies.get('session')?.value)

  if (isProtectedRoute && !session?.userId) {
    return NextResponse.redirect(new URL('/login', req.nextUrl))
  }

  return NextResponse.next()
}

export const config = {
  matcher: ['/((?!api|_next/static|_next/image|.*\\.png$).*)'],
}
```

**Key points**:
- Proxy uses the **Node.js runtime** by default in v16 (not Edge)
- Use `matcher` to exclude `_next/static`, `_next/image`, API routes, and static assets
- Proxy runs on **every route including prefetched routes** — only read from cookies (optimistic), avoid DB checks for performance
- **Negative matching** is required: exclude API routes, static files, and images explicitly

### Pattern B: Route Handler authentication (authoritative checks)

Always verify authentication inside the route handler itself — never rely on Proxy alone.

```ts
// app/api/route.ts
import { verifySession } from '@/app/lib/dal'

export async function GET() {
  const session = await verifySession()

  if (!session) {
    return new Response(null, { status: 401 })
  }

  if (session.user.role !== 'admin') {
    return new Response(null, { status: 403 })
  }

  // Continue for authorized users
}
```

### Critical Rule: Server Functions bypass Proxy matchers

> "Server Functions are not separate routes in this chain. They are handled as POST requests to the route where they are used, so a Proxy matcher that excludes a path will also skip Server Function calls on that path."
> — [Next.js Data Security Guide](https://nextjs.org/docs/app/guides/data-security)

**Always verify authentication and authorization inside each Server Function, not just in Proxy.**

## 2. Authorization Checking in Route Handlers

### DAL (Data Access Layer) Pattern

Create a centralized Data Access Layer that performs authorization checks close to data access:

```ts
// app/lib/dal.ts
import 'server-only'
import { cache } from 'react'
import { cookies } from 'next/headers'
import { decrypt } from '@/app/lib/session'

export const verifySession = cache(async () => {
  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)

  if (!session?.userId) {
    redirect('/login')
  }

  return { isAuth: true, userId: session.userId }
})
```

### Server Actions require re-authorization

A page-level auth check does NOT extend to Server Actions defined within it. Each Server Action must independently verify the user:

```tsx
// app/admin/page.tsx
export default async function AdminPage() {
  const session = await auth()
  if (!session?.user?.isAdmin) redirect('/login')

  return (
    <form action={async () => {
      'use server'
      const session = await auth()  // ← MUST re-verify here
      if (!session?.user?.isAdmin) throw new Error('Unauthorized')
      await db.record.deleteMany()
    }}>
      <button>Delete Records</button>
    </form>
  )
}
```

### Preventing IDOR (Insecure Direct Object Reference)

Always check resource ownership, not just authentication:

```ts
// app/actions.ts
'use server'
export async function deletePost(postId: string) {
  const session = await auth()
  if (!session?.user) throw new Error('Unauthorized')

  const post = await db.post.findUnique({ where: { id: postId } })

  // Check that the user owns this resource
  if (post.authorId !== session.user.id) {
    throw new Error('Forbidden')
  }

  await db.post.delete({ where: { id: postId } })
}
```

### Don't trust searchParams for authorization

```tsx
// BAD: Trusting searchParams directly
export default async function Page({ searchParams }) {
  const isAdmin = (await searchParams).isAdmin
  if (isAdmin === 'true') return <AdminPanel />  // Vulnerable!
}

// GOOD: Re-verify with cookies
export default async function Page() {
  const token = (await cookies()).get('AUTH_TOKEN')
  const isAdmin = await verifyAdmin(token)
  if (isAdmin) return <AdminPanel />
}
```

## 3. Rate Limiting Implementations

### Route Handler rate limiting

Next.js recommends adding rate limiting at the route handler level for expensive operations:

```ts
// app/api/resource/route.ts
import { NextResponse } from 'next/server'
import { checkRateLimit } from '@/lib/rate-limit'

export async function POST(request: Request) {
  const { rateLimited } = await checkRateLimit(request)

  if (rateLimited) {
    return NextResponse.json(
      { error: 'Rate limit exceeded' },
      { status: 429 }
    )
  }

  return new Response(null, { status: 204 })
}
```

### Implementation considerations

- Rate limit by IP address, user ID, or API key
- Apply to expensive operations: email sending, database writes, file uploads
- Also enable rate limiting features provided by your hosting platform (Vercel, Cloudflare, etc.)
- Consider using Redis-backed rate limiters for distributed environments

### Server Action rate limiting

Apply the same pattern inside Server Actions for mutation endpoints.

## 4. Input Validation and Sanitization

### Validate all client input

Never trust data from `searchParams`, dynamic params, form data, or request bodies. Always validate on the server:

```ts
import { z } from 'zod'

const SearchSchema = z.object({
  query: z.string().min(1).max(100).trim(),
  page: z.coerce.number().int().positive().default(1),
})

export async function GET(request: NextRequest) {
  const validated = SearchSchema.safeParse({
    query: request.nextUrl.searchParams.get('query'),
    page: request.nextUrl.searchParams.get('page'),
  })

  if (!validated.success) {
    return NextResponse.json(
      { errors: validated.error.flatten() },
      { status: 400 }
    )
  }

  // Use validated.data.query (sanitized)
}
```

### Validate dynamic route params

Folders with brackets (`[param]`) are user input — always validate:

```ts
export async function GET(
  request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params
  // Validate id is a valid format (e.g., UUID, number)
  if (!isValidId(id)) {
    return new Response(null, { status: 400 })
  }
}
```

### FormData validation with zod-form-data

Since `formData` values are all strings, use `zod-form-data` to validate and parse into correct types.

### Sanitize output

- Sanitize any input used to generate markup (prevents XSS)
- Don't return raw database records — return only what the client needs (DTOs)

## 5. Common Next.js API Security Vulnerabilities

### 5.1 Missing authorization in Server Actions

**Risk**: Server Actions are reachable via direct POST requests, not just through your UI.
**Fix**: Always verify authentication inside every Server Action, independently of page-level checks.

### 5.2 IDOR (Insecure Direct Object Reference)

**Risk**: User can access/modify resources they don't own by changing IDs.
**Fix**: Check resource ownership against the authenticated user.

### 5.3 Data over-exposure to Client Components

**Risk**: Passing entire database records as props to Client Components.
**Fix**: Use DTOs to return only necessary fields. Use `server-only` to prevent server modules from being imported client-side.

### 5.4 CSRF on Custom Route Handlers

**Risk**: Unlike Server Actions (which have built-in CSRF protection via Origin/Host header comparison), custom Route Handlers don't automatically protect against CSRF.
**Fix**: Implement CSRF token validation manually in `route.ts` for state-changing operations.

### 5.5 Open Redirects

**Risk**: Callback URLs or redirect parameters can be exploited for phishing.
**Fix**: Validate redirect destinations are same-origin:

```ts
const destination = new URL(redirectUrl ?? '/', request.url)
if (destination.origin !== request.nextUrl.origin) {
  return new Response('Invalid redirect', { status: 400 })
}
```

### 5.6 CORS misconfiguration

**Risk**: Wildcard `Access-Control-Allow-Origin: *` on sensitive routes.
**Fix**: Use Proxy with explicit origin allowlists for `/api` routes:

```ts
// proxy.ts
const allowedOrigins = ['https://acme.com', 'https://my-app.org']

export function proxy(request: NextRequest) {
  const origin = request.headers.get('origin') ?? ''
  const isAllowedOrigin = allowedOrigins.includes(origin)

  if (request.method === 'OPTIONS') {
    const preflightHeaders = {
      ...(isAllowedOrigin && { 'Access-Control-Allow-Origin': origin }),
      'Access-Control-Allow-Methods': 'GET, POST, PUT, DELETE, OPTIONS',
      'Access-Control-Allow-Headers': 'Content-Type, Authorization',
    }
    return NextResponse.json({}, { headers: preflightHeaders })
  }

  const response = NextResponse.next()
  if (isAllowedOrigin) {
    response.headers.set('Access-Control-Allow-Origin', origin)
  }
  return response
}
```

### 5.7 Sensitive data in error messages

**Risk**: Error messages may leak stack traces, credit card numbers, or internal details.
**Fix**: In production, Next.js/React replaces error messages with hashes. Always run in production mode for production workloads.

### 5.8 Closure variable exposure in Server Actions

**Risk**: Variables captured in Server Action closures are sent to the client and back.
**Fix**: Next.js encrypts closed-over variables automatically. New encryption key generated per build. For multi-server self-hosting, set `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY`.

### 5.9 Server-only code leakage

**Risk**: Server code accidentally imported by Client Components.
**Fix**: Use `import 'server-only'` in all server modules. Next.js will cause a build error if a Client Component imports this module.

## Summary of Recommended Architecture

1. **Proxy (`proxy.ts`)**: Optimistic auth checks, CORS, rate limiting at the edge
2. **Data Access Layer (`data/`)**: Centralized auth + authz logic, DTOs
3. **Route Handlers (`app/api/*/route.ts`)**: Public API endpoints with independent auth checks
4. **Server Actions (`'use server'`)**: Thin wrappers that delegate to DAL for auth/authz
5. **Use `server-only`**: In all DAL and server utility modules
6. **Validate with Zod**: All user inputs at system boundaries
