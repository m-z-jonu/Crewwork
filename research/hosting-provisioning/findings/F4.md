# F4: Stripe webhook + hosting provisioning integration patterns

## Findings

### [1] Stripe Entitlements is the purpose-built API for feature-based provisioning and de-provisioning
- quote: "Entitlements enable you to map the features of your internal service to Stripe products. After you map your features, Stripe notifies you about when to provision or de-provision access (according to your customer's subscription status), and to what features, based on your mapping choices."
- url: https://docs.stripe.com/billing/entitlements.md
- source_type: primary
- published: unknown
- confidence: high

### [2] The `entitlements.active_entitlement_summary.updated` webhook fires on every entitlement change including subscription creation, upgrades, downgrades, and cancellations
- quote: "When a customer's entitlements change because they subscribe to a product, upgrade, downgrade, or cancel, Stripe fires the `entitlements.active_entitlement_summary.updated` webhook. Use this event to drive access provisioning in your own system."
- url: https://docs.stripe.com/billing/entitlements.md
- source_type: primary
- published: unknown
- confidence: high

### [3] Revocation pattern: compare `entitlements.data` array before and after — removed features mean de-provision
- quote: "When a customer cancels their subscription or their subscription is canceled automatically due to failed payments, Stripe fires `entitlements.active_entitlement_summary.updated` again. At this point, the revoked features no longer appear in `entitlements.data`. Your application needs to disable the associated features or services accordingly."
- url: https://docs.stripe.com/billing/entitlements.md
- source_type: primary
- published: unknown
- confidence: high

### [4] Subscription status `canceled` and `unpaid` both indicate access should be revoked
- quote: "When a subscription changes to `canceled` or `unpaid`, revoke access to your product."
- url: https://docs.stripe.com/billing/subscriptions/webhooks
- source_type: primary
- published: unknown
- confidence: high

### [5] `invoice.paid` with active subscription status is the recommended event for initial provisioning
- quote: "You can provision access to your product when you receive this event and the subscription `status` is `active`."
- url: https://docs.stripe.com/billing/subscriptions/webhooks
- source_type: primary
- published: unknown
- confidence: high

### [6] Webhook handlers must return 2xx quickly; complex provisioning logic should be queued asynchronously
- quote: "Quickly return a successful status code (`2xx`) prior to any complex logic that might cause a timeout. For example, you must return a `200` response before updating a customer's invoice as paid in your accounting system."
- url: https://docs.stripe.com/webhooks
- source_type: primary
- published: unknown
- confidence: high

### [7] Stripe retries failed webhook deliveries for up to 3 days with exponential backoff in live mode (3 retries over hours in sandbox)
- quote: "Stripe attempts to deliver events to your destination for up to three days with an exponential back off in live mode."
- url: https://docs.stripe.com/webhooks
- source_type: primary
- published: unknown
- confidence: high

### [8] The recommended pattern is to store an access expiration timestamp and check it on login, rather than provisioning synchronously
- quote: "For typical integrations, you store customers' credentials and a mapped timestamp value that represents the access expiration date for that customer on your site when a customer subscribes. When the customer logs in, you check whether the timestamp is still in the future."
- url: https://docs.stripe.com/billing/subscriptions/webhooks
- source_type: primary
- published: unknown
- confidence: high

### [9] Features can be assigned lookup keys to map Stripe products to internal hosting account tiers (e.g., `premium-support`, `advanced-reporting`)
- quote: "Create each feature in Stripe Billing from the Dashboard. Here are some examples of features you can include: API access, AI assistant, Premium support, Advanced reporting, Extended data retention."
- url: https://docs.stripe.com/billing/entitlements.md
- source_type: primary
- published: unknown
- confidence: high

### [10] Webhook signature verification is mandatory to prevent attackers from triggering false provisioning
- quote: "Without verification, an attacker could send fake webhook events to your endpoint to trigger actions like fulfilling orders, granting account access, or modifying records."
- url: https://docs.stripe.com/webhooks
- source_type: primary
- published: unknown
- confidence: high

### [11] Duplicate events must be handled by logging processed event IDs to avoid double-provisioning
- quote: "Webhook endpoints might occasionally receive the same event more than once. You can guard against duplicated event receipts by logging the event IDs you've processed, and then not processing already-logged events."
- url: https://docs.stripe.com/webhooks
- source_type: primary
- published: unknown
- confidence: high

## Dead ends
- No third-party libraries specifically for "hosting provisioning via Stripe" found — the pattern is built from Stripe's Entitlements + Subscriptions webhooks combined with your own provisioning logic.
- Stripe Connect (SaaS platform pattern) is about managing payments for sub-merchants, not provisioning hosting accounts for end-users.

## Suggested follow-ups
- How to design the idempotent provisioning queue (e.g., using a state machine per customer with states like `provisioning`, `active`, `deprovisioning`)?
- What is the best way to handle the race condition between `customer.subscription.created` (trial start) and `invoice.paid` (payment confirmed) for initial provisioning?
- Should hosting provisioning use Entitlements-based webhooks or raw subscription status webhooks — what are the tradeoffs?
