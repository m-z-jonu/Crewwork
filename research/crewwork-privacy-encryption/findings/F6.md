# F6: Decentralized/P2P Alternatives

## Findings

### [1] Matrix is an open federated protocol where any homeserver can participate, with no single point of control or ownership over rooms
- quote: "Matrix defines a set of open APIs for decentralised communication, suitable for securely publishing, persisting and subscribing to data over a global open federation of servers with no single point of control."
- url: https://spec.matrix.org/latest/
- source_type: primary (official spec)
- published: 2026
- confidence: high

### [2] Matrix uses Olm (pairwise Double Ratchet) and Megolm (group ratchet) for end-to-end encryption, with spec v1.19 as latest
- quote: "Matrix uses the Olm and Megolm cryptographic ratchets for end-to-end encryption."
- url: https://spec.matrix.org/latest/olm-megolm/
- source_type: primary (official spec)
- published: 2026
- confidence: high

### [3] Matrix homeserver implementations include Synapse (Python, reference) and Conduit (Rust, lightweight), enabling self-hosting
- quote: "Conduit is a simple, fast and reliable chat server powered by Matrix"
- url: https://gitlab.com/famedly/conduit
- source_type: primary (official repo)
- published: 2020-present
- confidence: high

### [4] Session uses onion routing and a decentralized network of ~1,500+ nodes (each requiring 25,000 SESH stake), with no phone number required
- quote: "Each Session Node requires a stake of 25,000 SESH, and an attacker would need to control a substantial portion of the network's 1,500+ nodes for such an attack to be viable."
- url: https://getsession.org/blog/session-protocol-v2
- source_type: primary (official blog)
- published: 2025-12-01
- confidence: high

### [5] Session Protocol V2 is introducing Perfect Forward Secrecy (PFS) and Post-Quantum Cryptography (ML-KEM/CRYSTALS-Kyber) to address V1 limitations
- quote: "Session Protocol V2, a proposed upgrade to the Session Protocol which seeks to re-implement forward secrecy, utilizes new Post-Quantum Cryptography (PQC), and delivers improved device management features."
- url: https://getsession.org/blog/session-protocol-v2
- source_type: primary (official blog)
- published: 2025-12-01
- confidence: high

### [6] Session's V1 protocol lacks PFS because the Signal Protocol was deprecated due to multi-device sync failures in a decentralized network
- quote: "Session initially launched with PFS, inherited from the Signal Protocol, but the Signal Protocol was later deprecated in Session due to the significant issues it created for users with multiple linked devices."
- url: https://getsession.org/blog/session-protocol-v2
- source_type: primary (official blog)
- published: 2025-12-01
- confidence: high

### [7] Briar is a fully P2P messaging app using direct encrypted connections via Tor, with fallback to Bluetooth/Wi-Fi for offline operation
- quote: "Briar doesn't rely on a central server – messages are synchronized directly between the users' devices. If the Internet's down, Briar can sync via Bluetooth, Wi-Fi or memory cards."
- url: https://briarproject.org/how-it-works/
- source_type: primary (official docs)
- published: unknown
- confidence: high

### [8] Briar's threat model covers metadata surveillance (via Tor), content filtering (E2EE + no servers to block), takedown orders (distributed copies), and internet blackouts (Bluetooth/Wi-Fi)
- quote: "Briar provides private messaging, public forums and blogs that are protected against the following surveillance and censorship threats: Metadata surveillance, Content surveillance, Content filtering, Takedown orders, Denial of service attacks, Internet blackouts."
- url: https://briarproject.org/how-it-works/
- source_type: primary (official docs)
- published: unknown
- confidence: high

### [9] MLS (RFC 9420) is an IETF Internet Standard (published July 2023) for scalable group E2EE with forward secrecy and post-compromise security for groups of 2 to thousands
- quote: "In this document, we specify a key establishment protocol that provides efficient asynchronous group key establishment with forward secrecy (FS) and post-compromise security (PCS) for groups in size ranging from two to thousands."
- url: https://www.rfc-editor.org/rfc/rfc9420
- source_type: primary (IETF RFC)
- published: 2023-07
- confidence: high

### [10] MLS uses a tree-based key agreement structure (ratchet tree) where key update costs scale as log(N) of group size, enabling efficient large-group encryption
- quote: "This mechanism allows the members of the group to derive and update shared keys with costs that scale as the log of the group size."
- url: https://www.rfc-editor.org/rfc/rfc9420
- source_type: primary (IETF RFC)
- published: 2023-07
- confidence: high

### [11] Session faced severe financial constraints in early 2026 — the Session Technology Foundation paused development and needed community donations to resume
- quote: "Earlier this year, the Session Technology Foundation (STF) had to pause Session development due to financial constraints. After a donation campaign... the STF now has enough resources to continue Session development in some capacity."
- url: https://getsession.org/blog/the-future-of-session
- source_type: primary (official blog)
- published: 2026-06-15
- confidence: high

### [12] Nostr is a minimalist decentralized protocol using relays and public-key cryptography, but it has no built-in encryption — it requires layers like NIP-04 or NIP-59 for DMs
- quote: "No notes :( Please connect to relays" (Nostr homepage shows relay-dependent architecture with no inherent encryption layer)
- url: https://nostr.com/
- source_type: primary (official site)
- published: unknown
- confidence: medium
