# F3: Client-Side Encryption in JavaScript/TypeScript

## Findings

### [1] libsignal-protocol-javascript is deprecated and archived; libsignal-client with TypeScript API is the successor
- quote: "This library is no longer maintained. libsignal-protocol-javascript was an implementation of the Signal Protocol, written in JavaScript. It has been replaced by libsignal-client's typesafe TypeScript API."
- url: https://github.com/signalapp/libsignal-protocol-javascript
- source_type: primary
- published: 2021-08-05 (archived)
- confidence: high

### [2] Signal Protocol implements X3DH key agreement + Double Ratchet for forward secrecy, and now supports post-quantum SPQR ratchet as of October 2025
- quote: "Signal's implementation of the Sparse Post-Quantum Ratchet (SPQR), also known as t, in October 2025, demonstrates a leading-edge approach to securing communications against future threats."
- url: https://dasroot.net/posts/security/e2e-encryption-implementation-best-practices-for-2026
- source_type: secondary
- published: 2026-03-16
- confidence: high

### [3] WebCrypto API (crypto.subtle) is available in all modern browsers since July 2015, supports AES-GCM/CBC/CTR, RSA-OAEP, ECDH, ECDSA, Ed25519, PBKDF2, HKDF — but is Promise-based and lacks XChaCha20-Poly1305, Argon2, and Argon2id
- quote: "The Web Crypto API is an interface allowing a script to use cryptographic primitives in order to build systems using cryptography." ... "Baseline Widely available. This feature is well established and works across many devices and browser versions. It's been available across browsers since July 2015."
- url: https://developer.mozilla.org/en-US/docs/Web/API/Web_Crypto_API
- source_type: primary
- published: 2024-09-02
- confidence: high

### [4] CryptoKey objects stored in IndexedDB with extractable=false cannot be exported by JavaScript, but are still vulnerable to full device/OS-level access — there is no hardware-backed key storage on the web platform
- quote: "On the web platform, there is no equivalent to hardware-backed key storage; keys stored in IndexedDB are protected only by the browser's process isolation and the operating system's user account protections."
- url: https://www.askantech.com/home/end-to-end-encryption-in-modern-applications-implementation-without-the-mistakes
- source_type: secondary
- published: 2026-04-22
- confidence: high

### [5] noble-curves (v2.2.0) is an audited, tree-shakeable, pure-JS elliptic curve library — 32KB gzipped — supporting secp256k1, ed25519, ed448, p256/p384/p521, BLS12-381, X25519/X448, FROST threshold signatures, and includes a WebCrypto wrapper
- quote: "Audited & minimal JS implementation of elliptic curve cryptography. 🔒 Audited by independent security firms. 🔻 Tree-shakeable: unused code is excluded from your builds. 🪶 32KB (gzipped) including bundled hashes, 11KB for single-curve build."
- url: https://github.com/paulmillr/noble-curves
- source_type: primary
- published: 2026-04-12 (v2.2.0)
- confidence: high

### [6] noble-curves benchmarks on Apple M4: ed25519 sign ~6,849 ops/sec (145μs/op), verify ~1,400 ops/sec (713μs/op); x25519 ECDH ~1,981 ops/sec (504μs/op); secp256k1 sign ~7,182 ops/sec (139μs/op)
- quote: "ed25519 sign x 6,849 ops/sec @ 145μs/op, verify x 1,400 ops/sec @ 713μs/op, x25519 x 1,981 ops/sec @ 504μs/op"
- url: https://github.com/paulmillr/noble-curves
- source_type: primary
- published: 2026-04-12
- confidence: high

### [7] libsodium.js (WASM version) is 2–12x faster than noble-curves depending on operation; ed25519 sign is 6.6–7.2x faster, ed25519 verify is 6.9–12.3x faster, XChaCha20-Poly1305 encrypt is 2.3–4.0x faster — but total library size is 188KB gzipped vs noble's ~32KB
- quote: "Libsodium.js is faster in all cases except the random byte generation. The difference is in the one to two-digit range in terms of the factor." ... ed25519 sign: "10 Bytes — Noble 4,451 ops/sec, Libsodium 32,175 ops/sec, Factor 7.2"
- url: https://nikgraf.com/blog/choosing-a-cryptography-library-in-javascript-noble-vs-libsodium-js
- source_type: secondary
- published: 2024-05-21
- confidence: high

### [8] keystore-idb (Fission, TypeScript) manages ECC/RSA key pairs in IndexedDB via WebCrypto API, supports extractable=false keys, but ECC key storage in IndexedDB only works reliably in Chrome — Firefox and Safari fall back to RSA
- quote: "ECC (Elliptic Curve Cryptography) is only available on Chrome. Firefox and Safari do not support ECC and must use RSA. Specifically, this is an issue with storing ECC keys in IndexedDB."
- url: https://github.com/fission-codes/keystore-idb
- source_type: primary
- published: 2021-11-02 (v0.15.4)
- confidence: high

### [9] Frank Denis (libsodium author) recommends WebCrypto when possible and Noble for everything else — signalling a convergence of the JS crypto ecosystem toward these two pillars
- quote: "libsodium.js was a nice contribution, but honestly, for crypto in JavaScript today, I'd rather use WebCrypto when possible, and Noble cryptography for everything else."
- url: https://github.com/jedisct1/libsodium.js/issues/327#issuecomment-1793419292
- source_type: primary
- published: 2024
- confidence: high

### [10] WebCrypto API requires a secure context (HTTPS) and is not available in insecure contexts — keys stored with extractable=false cannot be serialized or accessed by JavaScript, providing protection against XSS exfiltration
- quote: "If the key is stored (as a cryptoKey object) in IndexedDB, and was not initiated as 'extractable' then it cannot be saved out from the browser. HOWEVER... That doesn't mean that the browser's data could not be copied out and compromised."
- url: https://gist.github.com/saulshanabrook/b74984677bccd08b028b30d9968623f5
- source_type: community
- published: unknown
- confidence: medium

### [11] For E2EE messaging in browsers, the core challenge is key management and trust verification, not the primitives — WebCrypto provides the primitives but no key exchange or identity verification protocol out of the box
- quote: "The thing that makes subtle crypto almost entirely useless is the lack of key management. Although you seem to use the primitives in the correct way, the key management is not specified at all in your question."
- url: https://crypto.stackexchange.com/questions/81609/sensible-amateur-usage-of-web-crypto-api
- source_type: community
- published: unknown
- confidence: high

### [12] @privacyresearch/libsignal-protocol-typescript (v0.0.16) is a TypeScript port of the Signal Protocol with a working demo, suitable for React+Node E2EE — but was last published in 2023 and is not actively maintained
- quote: "Signal Protocol Typescript implementation based on libsignal-protocol-javascript." ... "There are 7 other projects in the npm registry using @privacyresearch/libsignal-protocol-typescript."
- url: https://www.npmjs.com/package/@privacyresearch/libsignal-protocol-typescript
- source_type: primary
- published: 2023-05-06
- confidence: medium

## Dead ends
- `@aspect-build/rules_js` — this is a Bazel rules package for JS builds, not an encryption library; no relevant results.
- Google search was rate-limited (429) — all searches conducted via Brave Search instead.
- Signal Protocol in browser: Signal itself has no official web client because serving E2EE crypto from a server is fundamentally weaker (the server can modify JS at serve time, defeating E2EE). This is a known design limitation.
- "WebCrypto performance benchmarks" — no single canonical benchmark exists; performance varies by browser engine and hardware.
