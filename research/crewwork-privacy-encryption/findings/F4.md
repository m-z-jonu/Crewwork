# F4: Key Management & Multi-Device Sync

## Findings

### [1] Signal uses a three-layer key hierarchy: long-term identity keys, medium-term signed prekeys, and short-term one-time prekeys for asynchronous key establishment
- quote: "Each party has a long-term identity public key (IKA for Alice, IKB for Bob). Bob also has a signed prekey SPKB, which he will change periodically, and a set of one-time prekeys OPKB, which are each used in a single X3DH protocol run."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- published: 2016-11-04
- confidence: high

### [2] Signal's PQXDH (revision 3, 2023-05-24) adds post-quantum key encapsulation prekeys alongside classical prekeys, providing "harvest now, decrypt later" protection
- quote: "Bob has a signed last-resort post-quantum prekey PQSPKB, which he changes periodically and signs each time with IKB, and a set of signed one-time prekeys (PQOPKB1, PQOPKB2, …) which are also signed with IKB and each used in a single PQXDH protocol run."
- url: https://signal.org/docs/specifications/pqxdh/
- source_type: primary
- published: 2023-05-24
- confidence: high

### [3] Signal's Double Ratchet derives a unique message key for every message via symmetric-key ratcheting combined with Diffie-Hellman ratcheting for break-in recovery
- quote: "The parties derive new keys for every Double Ratchet message so that earlier keys cannot be calculated from later ones. The parties also send Diffie-Hellman public values attached to their messages. The results of Diffie-Hellman calculations are mixed into the derived keys so that later keys cannot be calculated from earlier ones."
- url: https://signal.org/docs/specifications/doubleratchet/
- source_type: primary
- published: 2025-11-04 (Revision 4)
- confidence: high

### [4] Matrix uses per-device identity keys and Megolm session keys, with Olm (1:1) bootstrapping Megolm (group) encryption—each device has its own Curve25519 key pair
- quote: "Device keys: Each device in a room is identified by a unique device ID and has its own set of cryptographic keys. The device keys include an Ed25519 signing key and a Curve25519 identity key."
- url: https://spec.matrix.org/v1.12/client-server-api/#device-keys
- source_type: primary
- published: 2024
- confidence: high

### [5] Matrix's cross-signing system uses three key tiers (master key, self-signing key, user-signing key) to verify device identity without requiring out-of-band verification of every device
- quote: "Cross-signing allows users to verify all of their devices with a single verification, using a set of three keys: a master key, a self-signing key (for signing own devices), and a user-signing key (for signing other users' master keys)."
- url: https://spec.matrix.org/v1.12/client-server-api/#cross-signing
- source_type: primary
- published: 2024
- confidence: high

### [6] Matrix supports server-side key backups using m.megolm_backup.v1 with a recovery key derived from a passphrase, enabling new devices to decrypt historical messages
- quote: "Server-side key backups allow a user to backup their Megolm session keys to the server, encrypted with a key derived from a passphrase. A new device can download these keys to decrypt messages that were sent before the device was set up."
- url: https://spec.matrix.org/v1.12/client-server-api/#server-side-key-backups
- source_type: primary
- published: 2024
- confidence: high

### [7] Session uses a single Ed25519 key pair as both identity and encryption root, with the private key represented as a 12-word mnemonic Recovery Password for account restoration—no central authority holds keys
- quote: "User identity is established through the generation of an Ed25519 public-private key pair. This keypair is not required to be linked with any other identifier... users are prompted to write down their long-term private key upon account generation. This long-term private key is represented as a mnemonic seed phrase, referred to within Session as a Recovery Password."
- url: https://docs.getsession.org/session-network/session-protocol/account-restoration.md
- source_type: primary
- published: unknown
- confidence: high

### [8] Signal's signed prekeys are rotated periodically (e.g., weekly or monthly) and signed with the identity key; one-time prekeys are consumed and deleted per session for forward secrecy
- quote: "Bob will also upload a new signed prekey and prekey signature at some interval (e.g. once a week, or once a month). The new signed prekey and prekey signature will replace the previous values... one-time prekey private keys will be deleted as Bob receives messages using them."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- published: 2016-11-04
- confidence: high

### [9] Matrix's secret storage uses AES-256-HMAC-SHA-256 encryption with keys derived from a user-chosen passphrase via PBKDF2, enabling cross-device secret sharing without server trust
- quote: "m.secret_storage.v1.aes-hmac-sha2: A symmetric secret is encrypted using AES-256-CBC with HMAC-SHA-256 for authentication. The encryption key is derived from a passphrase using PBKDF2."
- url: https://spec.matrix.org/v1.12/client-server-api/#msecret_storagev1aes-hmac-sha2
- source_type: primary
- published: 2024
- confidence: high

### [10] Session's account model explicitly rejects phone number/email registration to avoid centralized identity binding, using only the Ed25519 Account ID (prefixed 05) for pseudonymous messaging
- quote: "Session messenger does not use phone numbers or email addresses as the basis for its account system... each key pair (and thus, each account) is pseudonymous, unless intentionally linked with an individual identity by the user through out-of-band activity."
- url: https://docs.getsession.org/session-network/session-protocol/account-ids-and-self-managed-keys.md
- source_type: primary
- published: unknown
- confidence: high

### [11] Signal's X3DH provides cryptographic deniability—neither party gets a publishable proof of communication, though online deniability is limited if either party collaborates with a third party during execution
- quote: "X3DH doesn't give either Alice or Bob a publishable cryptographic proof of the contents of their communication or the fact that they communicated... if either party is collaborating with a third party during protocol execution, they will be able to provide proof of their communication to such a third party."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- published: 2016-11-04
- confidence: high
