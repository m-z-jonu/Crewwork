# F8: Practical Implementation Roadmap

## Findings

### [1] PowerSync + Supabase E2EE chat demo proves the encrypted-mirror pattern works in production: store opaque cipher envelopes in Supabase, decrypt locally, expose plaintext via mirror tables for the UI
- quote: "Messages and rooms are stored as encrypted rows; mirrors expose the plaintext needed for UI. I used PowerSync's differential watch to feed the chat, because it tells you exactly what changed (added/updated/removed) rather than re-emitting whole result sets."
- url: https://powersync.com/blog/building-an-e2ee-chat-app-with-powersync-supabase
- source_type: primary (author built the system)
- published: 2025-10-08
- confidence: high

### [2] PowerSync E2EE monorepo implements four reusable crypto packages: interface types, encrypted-sqlite runtime, password-based PBKDF2 provider, and WebAuthn-based passkey provider — all keeping ciphertext opaque to the backend
- quote: "packages/crypto/encrypted-sqlite — Encrypted↔mirror runtime for SQLite/PowerSync: pair configs, ensurePairsDDL, mirror orchestration, and CRUD helpers. packages/crypto/password — Password-based crypto provider (PBKDF2 by default) implementing the CryptoProvider interface for wrapping/unwrapping DEKs. packages/crypto/webauthn — WebAuthn-based crypto provider using PRF/hmac-secret extensions to derive a stable secret per credential."
- url: https://github.com/powersync-community/react-supabase-chat-e2ee
- source_type: primary (open-source reference implementation)
- published: unknown
- confidence: high

### [3] The cipher envelope pattern uses a standard structure: {alg, aad, nonce, ciphertext, KDF params} stored as separate columns, which is the recommended approach for incremental E2EE on existing Supabase tables
- quote: "What we sync is an opaque cipher envelope (alg, aad, nonce, ciphertext, KDF params), not a JSON business object. The JSON view model can't decrypt on the client's behalf or project fields out of ciphertext."
- url: https://powersync.com/blog/building-an-e2ee-chat-app-with-powersync-supabase
- source_type: primary
- published: 2025-10-08
- confidence: high

### [4] For raw tables (needed for E2EE), adding columns requires client-side full re-sync since there's no incremental column migration mechanism; server data stays intact but local data must be wiped for affected tables
- quote: "If you add columns to a raw table, there's currently no 're-sync with new columns' mechanism. The guidance is to delete local data for that table and let the client perform a full re-sync so the new columns populate. This is a client-side reset only — server data stays intact."
- url: https://powersync.com/blog/building-an-e2ee-chat-app-with-powersync-supabase
- source_type: primary
- published: 2025-10-08
- confidence: high

### [5] Supabase E2EE chat by arnu515 demonstrates a simpler pattern using TweetNaCl for encryption, Supabase Realtime for message delivery, and Dexie.js (IndexedDB) for local storage — both users must be online since messages are NOT stored on server
- quote: "Both users have to be online to receive chat messages. This is because the chat messages are NOT stored on the server. They are stored in the browser's local IndexedDB storage."
- url: https://github.com/arnu515/supabase-e2ee-chat
- source_type: community (hackathon winner, 35 stars)
- published: unknown
- confidence: medium

### [6] CipherStash Stack provides field-level encryption for Supabase with searchable queries — encrypted columns support equality, LIKE, ranges, and ordering over ciphertext using EQL (Encrypt Query Language) types installed as a Postgres extension
- quote: "CipherStash for Supabase: Official Security Partner — Field-level encryption, searchable queries, and identity-bound keys to your Supabase project today."
- url: https://cipherstash.com/supabase
- source_type: primary (commercial product docs)
- published: unknown
- confidence: high

### [7] CipherStash's encryptedSupabase wrapper maintains API compatibility with the standard Supabase JS client — you pass a schema argument to .from() and values are encrypted/decrypted transparently, making migration incremental column-by-column
- quote: "encryptedSupabase() returns a client with the same API as the regular Supabase JS client... with an extra users schema argument and no select('*'). The values for .eq(), .ilike(), .gt() and so on are encrypted automatically before the query reaches Postgres; results are decrypted automatically on the way back."
- url: https://cipherstash.com/blog/encrypting-supabase-data-with-cipherstash-stack
- source_type: primary
- published: 2026-05-24
- confidence: high

### [8] The libsignal-protocol-javascript library provides a complete Signal Protocol implementation for browsers but is deprecated — the TypeScript interface of libsignal-client is recommended for all new applications; it uses Emscripten-compiled C for elliptic curve ops and WebCrypto for symmetric crypto
- quote: "This library is deprecated. The TypeScript interface of libsignal-client is recommended for all new applications."
- url: https://deepwiki.com/signalapp/libsignal-protocol-javascript
- source_type: primary (Signal Foundation official)
- published: 2025-06-25 (indexed date)
- confidence: high

### [9] The Web Crypto API (available in all browsers since July 2015) provides AES-GCM, PBKDF2, ECDH, HKDF, and RSA-OAEP — these are the primitives needed for client-side E2EE without any third-party libraries
- quote: "The Web Crypto API is an interface allowing a script to use cryptographic primitives in order to build systems using cryptography."
- url: https://developer.mozilla.org/en-US/docs/Web/API/Web_Crypto_API
- source_type: primary (MDN, W3C spec)
- published: 2024-09-02
- confidence: high

### [10] Supabase Vault stores secrets using Authenticated Encryption with Associated Data (AEAD) based on libsodium — the encryption key is never stored in the database and is managed in Supabase's secured backend systems
- quote: "The encryption key is never stored in the database alongside the encrypted data. Even if an attacker can capture a dump of your entire database, they will see only encrypted data, never the encryption key itself."
- url: https://supabase.com/docs/guides/database/vault
- source_type: primary (Supabase official docs)
- published: unknown
- confidence: high

### [11] pgsodium is pending deprecation at Supabase — Supabase does NOT recommend any new usage; the Vault extension's internal implementation will shift away from pgsodium but its interface remains unchanged; projects encrypted at rest by default may already satisfy SOC2 & HIPAA compliance
- quote: "Supabase DOES NOT RECOMMEND any new usage of pgsodium. The pgsodium extension is expected to go through a deprecation cycle in the near future."
- url: https://supabase.com/docs/guides/database/column-encryption
- source_type: primary (Supabase official docs)
- published: unknown
- confidence: high

### [12] TipTap stores content as either JSON (ProseMirror document) or HTML — both formats are equally suitable for encryption; JSON is easier to manipulate programmatically and is what TipTap uses internally, making it the better candidate for encrypt-then-store
- quote: "You can store your content as a JSON object or as a good old HTML string... JSON is probably easier to loop through, for example to look for a mention and it's more like what Tiptap uses under the hood."
- url: https://tiptap.dev/docs/editor/guide/output#html
- source_type: primary (TipTap official docs)
- published: unknown
- confidence: high
