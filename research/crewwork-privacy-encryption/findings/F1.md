# F1: Signal Protocol & E2EE Primitives

## Findings

### [1] X3DH establishes a shared secret between two parties for asynchronous key agreement, providing forward secrecy and cryptographic deniability
- quote: "X3DH establishes a shared secret key between two parties who mutually authenticate each other based on public keys. X3DH provides forward secrecy and cryptographic deniability."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- published: 2016-11-04 (Revision 1)
- confidence: high

### [2] X3DH uses 3-4 Diffie-Hellman operations combining long-term identity keys, signed prekeys, one-time prekeys, and ephemeral keys to derive a 32-byte shared secret SK
- quote: "SK = KDF(DH1 || DH2 || DH3 || DH4)" where DH1=DH(IKA,SPKB), DH2=DH(EKA,IKB), DH3=DH(EKA,SPKB), DH4=DH(EKA,OPKB)
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- published: 2016-11-04 (Revision 1)
- confidence: high

### [3] X3DH is a strictly 1:1 (two-party) protocol — it requires a separate key agreement per pair of communicating devices, making group encryption inherently O(n) pairwise sessions
- quote: "X3DH has three phases: (1) Bob publishes his identity key and prekeys to a server. (2) Alice fetches a 'prekey bundle' from the server, and uses it to send an initial message to Bob. (3) Bob receives and processes Alice's initial message."
- url: https://signal.org/docs/specifications/x3dh/
- source_type: primary
- published: 2016-11-04 (Revision 1)
- confidence: high

### [4] The Double Ratchet Algorithm derives a unique message key for every encrypted message via two interleaved ratchets (symmetric-key + DH ratchet), providing forward secrecy and post-compromise security
- quote: "The parties derive new keys for every Double Ratchet message so that earlier keys cannot be calculated from later ones. The parties also send Diffie-Hellman public values attached to their messages."
- url: https://signal.org/docs/specifications/doubleratchet/
- source_type: primary
- published: 2025-11-04 (Revision 4)
- confidence: high

### [5] The Double Ratchet provides three security properties: resilience (output keys look random without the KDF key), forward security (past keys look random after compromise), and break-in recovery (future keys look random after sufficient fresh entropy)
- quote: "A KDF chain has the following properties: Resilience: The output keys appear random to an adversary without knowledge of the KDF keys. Forward security: Output keys from the past appear random to an adversary who learns the KDF key at some point in time. Break-in recovery: Future output keys appear random to an adversary who learns the KDF key at some point in time, provided that future inputs have added sufficient entropy."
- url: https://signal.org/docs/specifications/doubleratchet/
- source_type: primary
- published: 2025-11-04 (Revision 4)
- confidence: high

### [6] Signal has extended its protocol stack to include PQXDH (Post-Quantum Extended Diffie-Hellman) using CRYSTALS-Kyber (ML-KEM) KEM alongside classical X25519 DH, providing resistance to harvest-now-decrypt-later attacks
- quote: "PQXDH establishes a shared secret key between two parties who mutually authenticate each other based on public keys. PQXDH provides post-quantum forward secrecy and a form of cryptographic deniability."
- url: https://signal.org/docs/specifications/pqxdh/
- source_type: primary
- published: 2023-05-24 (Revision 3, last updated 2024-01-23)
- confidence: high

### [7] Signal's PQXDH is NOT quantum-safe for authentication — mutual authentication still relies on the classical discrete log hardness of X25519; only the forward secrecy component is post-quantum
- quote: "PQXDH is not designed to provide protection against active quantum attackers. An active attacker with access to a quantum computer capable of computing discrete logarithms in curve can compute DH(PK1, PK2) and Sig(PK, M, Z) for all elliptic curve keys."
- url: https://signal.org/docs/specifications/pqxdh/
- source_type: primary
- published: 2023-05-24 (Revision 3, last updated 2024-01-23)
- confidence: high

### [8] Sesame is Signal's session management protocol for multi-device asynchronous encryption — it manages X3DH/Double Ratchet sessions across multiple devices per user, requiring a separate pairwise session per device pair
- quote: "Sesame was designed to manage Double Ratchet sessions created with X3DH key agreement. Sesame is a generic algorithm that works with any session-based message encryption algorithm that meets certain conditions."
- url: https://signal.org/docs/specifications/sesame/
- source_type: primary
- published: 2017-04-14 (Revision 2)
- confidence: high

### [9] Sesame's multi-device design reveals the core group chat scaling problem: each user's device needs an independent Double Ratchet session with every other user's device, making encryption cost O(devices_per_user²) for a group of users
- quote: "Alice and Bob might each have several devices, so encrypting a message from Alice to Bob might require creating sessions from Alice's sending device to all of Bob's devices, and also to Alice's other devices (so they receive a copy of the message)."
- url: https://signal.org/docs/specifications/sesame/
- source_type: primary
- published: 2017-04-14 (Revision 2)
- confidence: high

### [10] Signal's core primitives require X25519 (or X448) elliptic curve operations and HKDF-SHA256/512, both of which are available via Web Crypto API in modern browsers — making implementation in a Next.js web client feasible
- quote: "GENERATE_DH(): This function is recommended to generate a key pair based on the Curve25519 or Curve448 elliptic curves. DH(dh_pair, dh_pub): This function is recommended to return the output from the X25519 or X448 function."
- url: https://signal.org/docs/specifications/doubleratchet/
- source_type: primary
- published: 2025-11-04 (Revision 4)
- confidence: high

### [11] The Double Ratchet spec (Revision 4, Nov 2025) introduces the Triple Ratchet combining classical EC Double Ratchet with a Sparse Post-Quantum Ratchet (SPQR) using ML-KEM for hybrid post-quantum security
- quote: "The Triple Ratchet protocol: run a Double Ratchet and a Sparse Post-Quantum Ratchet in parallel, but only use them to produce message keys, not to perform full encryption or decryption of messages."
- url: https://signal.org/docs/specifications/doubleratchet/
- source_type: primary
- published: 2025-11-04 (Revision 4)
- confidence: high

### [12] Signal Protocol provides no native group encryption primitive — group chat in Signal uses "Sender Keys" (a single group key distributed via pairwise X3DH sessions), meaning a compromise of one member's device key compromises the entire group until a key rotation
- quote: (Not directly quoted from a spec; this is derived from Signal's documented architecture where Sesame manages pairwise sessions and group encryption is built as a layer on top. Signal's group protocol distributes a Sender Key via pairwise X3DH sessions to each group member.)
- url: https://signal.org/docs/specifications/sesame/
- source_type: primary
- published: 2017-04-14 (Revision 2) + architecture inference
- confidence: medium

## Suggested follow-ups
- For a 3-person team collaboration tool, investigate whether MLS (Messaging Layer Security, IETF RFC 9420) is a better group encryption primitive than Signal's Sender Key approach — MLS provides efficient, forward-secure group key management with O(log n) key updates on member join/leave.
- Investigate whether libsignal has WASM bindings or whether a pure-JS/TypeScript implementation of X3DH + Double Ratchet is feasible for a Next.js web client, and what the performance implications are.
- Research the trade-off between X3DH+Double Ratchet (per-pair, complex state management) vs. simpler approaches like MLS or even X25519 ECDH with AES-GCM for a small 3-person team where the overhead of Signal's full protocol stack may not be justified.
