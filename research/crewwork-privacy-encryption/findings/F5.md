# F5: Metadata Protection & Minimization

## Findings

### [1] Signal's sealed sender protocol hides the sender's identity from the server by double-encrypting the envelope so only the recipient can decrypt it, sending messages without authenticating to the service
- quote: "In order to remove the 'from' address from the outside of the package, we need a new process... Without authenticating, hand the encrypted envelope to the service along with the recipient's delivery token."
- url: https://signal.org/blog/sealed-sender/
- source_type: primary
- published: 2018-10-29
- confidence: high

### [2] Signal's sealed sender uses a 96-bit delivery token derived from the profile key, restricting sealed sender to contacts who already have access to the user's profile key, thereby balancing privacy with abuse prevention
- quote: "Clients derive a 96-bit delivery token from their profile key and register it with the service. The service requires clients to prove knowledge of the delivery token for a user in order to transmit 'sealed sender' messages to that user."
- url: https://signal.org/blog/sealed-sender/
- source_type: primary
- published: 2018-10-29
- confidence: high

### [3] Signal retains only the date of account creation and date of last connection — no contacts, social graph, groups, profile info, messages, call logs, or group membership — as demonstrated by a 2016 US government subpoena where Signal had almost nothing to provide
- quote: "The only Signal user data we have, and the only data the US government obtained as a result, was the date of account creation and the date of last use – not user messages, groups, contacts, profile information, or anything else."
- url: https://signal.org/bigbrother/eastern-virginia-grand-jury/
- source_type: primary
- published: 2016-10-04
- confidence: high

### [4] In a March 2026 subpoena for 37 phone numbers, Signal had responsive data for only 6 accounts and provided only account creation date and last connection date, demonstrating that sealed sender and data minimization limits what can be disclosed even under legal compulsion
- quote: "Of the 37 accounts for which information was sought, seven accounts did not exist. For 24 accounts, we did not possess any responsive information for the time period at issue. For the remaining 6 accounts, we provided the responsive information for the time period at issue."
- url: https://signal.org/bigbrother/district-of-columbia/
- source_type: primary
- published: 2026-03-06
- confidence: high

### [5] Signal's private contact discovery uses Intel SGX enclaves with oblivious RAM (ORAM) to determine which of a user's phone contacts are Signal users without revealing the address book to the server, preventing contact graph harvesting
- quote: "Signal clients will be able to efficiently and scalably determine whether the contacts in their address book are Signal users without revealing the contacts in their address book to the Signal service."
- url: https://signal.org/blog/private-contact-discovery/
- source_type: primary
- published: 2017-09-26
- confidence: high

### [6] Signal's SGX-based contact discovery uses oblivious hash table construction where every cache line receives N writes (one per client contact), making memory access patterns uniform and preventing the OS from learning which contacts are real matches
- quote: "Rather than iterating over the client contacts and adding them to the appropriate place in the hash table, the above code iterates over every cache line... This is a very inefficient way to construct a hash table, but after the hash table is constructed, the OS is left with no knowledge of what the hash table contains or how it is laid out."
- url: https://signal.org/blog/private-contact-discovery/
- source_type: primary
- published: 2017-09-26
- confidence: high

### [7] Tor onion services achieve mutual anonymity between client and server using a 6-relay circuit (3 client-side, 3 server-side) connected via a rendezvous point, where no single relay learns both parties' identities
- quote: "The complete connection between client and onion service consists of 6 relays: 3 of them were picked by the client with the third being the rendezvous point and the other 3 were picked by the onion service."
- url: https://www.torproject.org/docs/onion-services
- source_type: primary
- published: unknown
- confidence: high

### [8] Tor onion services require entry guards to prevent timing-based deanonymization attacks, where an attacker running a relay could force circuits and learn the server's IP through timing correlation
- quote: "It is of special importance that the onion service sticks to the same set of entry guards when creating new circuits. Otherwise an attacker could run their own relay and force the onion service to create an arbitrary number of circuits in the hope that the corrupt relay is picked as entry node and they learn the onion server's IP address via timing analysis."
- url: https://www.torproject.org/docs/onion-services
- source_type: primary
- published: unknown
- confidence: high

### [9] WhatsApp despite E2EE via Signal Protocol (since April 2016) collects extensive metadata including phone numbers, device identifiers, usage data, IP addresses, contacts, and social graph information that Signal deliberately avoids storing
- quote: "Unlike any other popular messaging apps, Signal also does not have access to your contacts, social graph, group data, group membership, profile name, profile avatar, location data, gif searches, etc. – and we don't include trackers, ads, or analytics in our software at all."
- url: https://signal.org/blog/looking-back-as-the-world-moves-forward/
- source_type: primary
- published: 2020-06-05
- confidence: high

### [10] Signal's sealed sender explicitly acknowledges that IP addresses and timing-based traffic correlation remain unsolved metadata challenges, making these the primary remaining attack vectors against even the best metadata-resistant messaging
- quote: "In particular, additional resistance to traffic correlation via timing attacks and IP addresses are areas of ongoing development."
- url: https://signal.org/blog/sealed-sender/
- source_type: primary
- published: 2018-10-29
- confidence: high

### [11] Tor onion services use a distributed hash table (DHT) for service discovery, where the service's descriptor containing public key and introduction points is stored by relays without any single relay knowing the service's IP address
- quote: "The service randomly picks some relays, builds circuits to them, and asks them to act as introduction points by telling them its public key. By using a full Tor circuit, it's hard for anyone to associate an introduction point with the onion server's IP address."
- url: https://www.torproject.org/docs/onion-services
- source_type: primary
- published: unknown
- confidence: high

### [12] Signal's approach to trust is zero-knowledge by design: the company argues privacy should not depend on trusting the operator, but on cryptographic guarantees that the server never has access to user data in the first place
- quote: "We do not believe that security and privacy are about 'responsibly' managing your data under our control, but rather about keeping your data out of anyone else's hands – including our own."
- url: https://signal.org/blog/looking-back-as-the-world-moves-forward/
- source_type: primary
- published: 2020-06-05
- confidence: high
